# Constitution: 사용자 채용 공고

**Feature Branch**: `feature/user-job-notice`
**Scope**: 채용 공고 목록 페이지, 채용 공고 상세 드로어 공통

## 1. 도메인 가치

구직자는 Career Wave 안에서 여러 출처의 채용 공고를 검색, 필터링, 비교하고 원본 공고로 이동할 수 있어야 한다.
목록은 탐색 속도와 조건 축소를 우선하고, 상세는 지원 전 확인해야 할 핵심 채용 조건과 기업 정보를 신뢰성 있게 제공한다.
스크랩 상태는 구직자의 개인 액션이므로 목록, 상세, 마이페이지에서 항상 같은 결과로 보인다.

## 2. 상태 머신

### 목록 페이지

```text
초기 진입
  -> 목록 로딩
    -> 목록 표시
    -> 빈 결과
    -> 오류

목록 표시
  -> 검색/필터/기간/정렬 변경
    -> 목록 로딩
  -> 카드 선택
    -> 상세 열림
  -> 스크랩 클릭
    -> 스크랩 저장/해제
```

| 전이 | 허용 여부 | 사유 |
|------|-----------|------|
| 목록 표시 -> 검색/필터/기간/정렬 변경 | 허용 | 사용자가 탐색 조건을 바꾸면 목록 결과를 다시 계산하거나 조회한다. |
| 빈 결과 -> 검색/필터 변경 | 허용 | 빈 결과 상태에서도 조건 완화가 가능해야 한다. |
| 오류 -> 재시도 | 허용 | 네트워크 또는 서버 장애 복구를 지원한다. |
| 비로그인 -> 스크랩 저장 | 조건부 허용 | 로그인 필요 안내 후 로그인 페이지로 유도한다. |

### 상세 드로어

```text
닫힘
  -> 열림/상세 로딩
    -> 상세 표시
    -> 상세 빈 상태
    -> 상세 오류
  -> 닫힘
```

| 전이 | 허용 여부 | 사유 |
|------|-----------|------|
| 닫힘 -> 열림/상세 로딩 | 허용 | 목록 카드 선택 또는 외부 딥링크로 상세를 열 수 있다. |
| 상세 표시 -> 닫힘 | 허용 | 닫기 버튼, 오버레이 클릭, Escape 키를 지원한다. |
| 상세 표시 -> 스크랩 저장/해제 | 허용 | 상세에서도 목록과 동일한 스크랩 액션을 제공한다. |

## 3. 아키텍처 결정

| 결정 | 내용 | 근거 |
|------|------|------|
| 공통 타입 | 목록 카드와 상세 드로어는 `JobNotice` 계열 타입을 공유한다. | 같은 공고의 표시 값과 스크랩 상태가 화면마다 달라지는 문제를 막는다. |
| 목록 조회 계약 | 검색어, 직무, 경력, 채용 유형, 지역, 기업 규모, 기간, 정렬은 API 쿼리 파라미터로 표현한다. | 서버 페이지네이션 전환 시에도 프론트엔드 상태 모델을 유지할 수 있다. |
| 상세 표시 방식 | 목록 페이지 안에서 `JobNoticeDetail` 드로어를 열고 닫는다. | 공고 탐색 맥락을 유지하면서 상세 정보를 빠르게 확인하게 한다. |
| 스크랩 처리 | 스크랩은 낙관적 UI 업데이트를 허용하되 실패 시 이전 상태로 되돌린다. | 버튼 반응성은 유지하면서 서버의 최종 상태를 기준으로 정합성을 맞춘다. |
| 원본 공고 이동 | 상세의 `originalUrl`은 새 탭으로 연다. | Career Wave 탐색 흐름을 보존하고 원본 사이트 확인을 가능하게 한다. |
| API 호출 계층 | 목록/상세/스크랩 API 호출은 `frontend/src/user/api` 하위 도메인 API 모듈을 통해 수행한다. | 페이지 또는 컴포넌트에서 `axios`나 `fetch`를 직접 호출하지 않는 팀 규칙을 지킨다. |
| 서버 상태 관리 | API 연동 후 목록, 상세, 스크랩 mutation은 TanStack Query 사용을 우선한다. | `CONVENTION.md`의 Frontend State Management 원칙을 따른다. |

## 4. 불변 규칙

- 목록과 상세에 노출되는 공고는 공개 상태이며 마감 또는 비공개 공고는 별도 안내 없이 신규 목록에 포함하지 않는다.
- 동일한 공고 `id`의 `bookmarked` 값은 목록 카드, 상세 드로어, 스크랩 공고 페이지에서 동일해야 한다.
- 검색어는 회사명, 공고명, 직무, 기술 스택, 태그, 출처를 대상으로 하며 공백과 대소문자 차이 때문에 결과가 누락되면 안 된다.
- 경력 범위 필터는 교집합 기준으로 판단한다. 예를 들어 `3~5년` 조건은 `1~3년`, `3~20년` 공고와 경계값이 겹치면 포함될 수 있다.
- 필터의 `전체` 값은 해당 조건을 적용하지 않는다는 뜻이며 API 요청에서는 생략할 수 있다.
- 상세에서 원본 공고로 이동하기 전까지 Career Wave 내부 지원 상태는 변경하지 않는다.
- 목록과 상세는 사용자가 이해할 수 있는 한국어 라벨을 사용하고, 데이터 원천의 원문 문자열이 깨진 상태로 화면에 노출되면 안 된다.

## 5. 연동 계약

- `user-backend`는 `api-spec.md`의 목록 조회, 상세 조회, 스크랩 토글 계약을 제공한다.
- 모든 Backend API 응답은 `ApiResponse<T>` 형식(`success`, `statusCode`, `message`, `data`)을 사용한다.
- 예상 가능한 실패는 `ErrorCode`와 공통 예외 흐름을 통해 처리하고, 프론트엔드는 `message`를 사용자 안내 문구로 활용한다.
- `admin-backend` 또는 수집 파이프라인에서 비공개, 반려, 마감 처리한 공고는 사용자 목록 API에 노출하지 않는다.
- FastAPI 수집 데이터는 사용자에게 노출되기 전 회사명, 제목, 원본 URL, 직무, 지역, 마감 정보를 정규화해야 한다.
- 마이페이지 스크랩 공고는 같은 스크랩 저장소를 사용하며 목록/상세의 토글 결과를 반영한다.
- 지원하기, AI 면접, 대시보드 추천 공고 등 다른 사용자 기능에서 공고를 식별할 때는 동일한 공고 `id`를 사용한다.

## 6. 금지 패턴

- 목록과 상세가 서로 다른 mock 데이터를 따로 보관하는 것을 금지한다.
- 공고 ID 대신 배열 index로 상세 선택, 스크랩 상태, 라우팅을 처리하는 것을 금지한다.
- API 실패 시 조용히 빈 목록으로 바꾸는 것을 금지한다. 빈 결과와 오류는 사용자 메시지와 재시도 가능 여부가 달라야 한다.
- 스크랩 API 실패 후 UI 상태를 성공처럼 유지하는 것을 금지한다.
- 상세 드로어가 열린 상태에서 배경 스크롤이 계속 동작하거나 Escape 키로 닫을 수 없는 상태를 금지한다.
- `any` 타입으로 API 응답을 통과시키고 화면에서 직접 필드를 추측하는 것을 금지한다.
- 페이지 또는 컴포넌트에서 API 클라이언트를 직접 구현하거나 `axios`를 직접 호출하는 것을 금지한다.
