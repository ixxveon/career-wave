# Feature Specification: 채용 공고 리스트

**Feature Branch**: `feature/user-job-notice-list`
**Status**: Draft

## User Scenarios & Testing

### User Story 1 - 공고 목록 탐색 (Priority: P1)

> 구직자는 채용 공고 목록에서 현재 공개된 공고를 카드 형태로 확인하고, 원하는 공고를 빠르게 찾을 수 있다.

**Acceptance Scenarios**:
1. **Given** 공개된 채용 공고가 존재할 때, **When** 사용자가 `/jobs` 목록 페이지에 진입하면, **Then** 공고 카드 목록과 전체 공고 통계가 표시된다.
2. **Given** 목록에 여러 공고가 있을 때, **When** 사용자가 공고 카드를 선택하면, **Then** 선택한 공고의 상세 드로어가 열린다.
3. **Given** 검색어 입력창이 표시되어 있을 때, **When** 사용자가 회사명, 공고명, 직무, 기술 스택 중 하나를 입력하고 검색하면, **Then** 해당 키워드와 일치하는 공고만 표시된다.

---

### User Story 2 - 조건 필터링과 정렬 (Priority: P1)

> 구직자는 직무, 경력, 채용 유형, 지역, 기업 규모, 게시 기간, 정렬 기준을 조합하여 공고 목록을 좁힐 수 있다.

**Acceptance Scenarios**:
1. **Given** 사용자가 직무 필터에서 `백엔드`를 선택했을 때, **When** 목록이 갱신되면, **Then** 백엔드 직무와 관련된 공고만 표시된다.
2. **Given** 사용자가 여러 필터를 선택했을 때, **When** 활성 필터 칩을 확인하면, **Then** `전체`가 아닌 조건만 칩으로 표시된다.
3. **Given** 활성 필터 칩이 표시되어 있을 때, **When** 사용자가 칩을 클릭하면, **Then** 해당 필터만 `전체`로 초기화되고 목록이 다시 갱신된다.
4. **Given** 사용자가 `최신순` 또는 `조회순`을 선택했을 때, **When** 정렬 메뉴가 닫히면, **Then** 선택한 기준으로 공고 카드 순서가 변경된다.

---

### User Story 3 - 스크랩과 개인화 액션 (Priority: P2)

> 구직자는 관심 있는 공고를 목록에서 바로 스크랩하거나 해제할 수 있다.

**Acceptance Scenarios**:
1. **Given** 로그인한 사용자가 스크랩되지 않은 공고를 보고 있을 때, **When** 북마크 버튼을 클릭하면, **Then** 해당 공고는 스크랩 상태로 표시된다.
2. **Given** 스크랩된 공고가 표시되어 있을 때, **When** 사용자가 다시 북마크 버튼을 클릭하면, **Then** 해당 공고의 스크랩이 해제된다.
3. **Given** 비로그인 사용자가 북마크 버튼을 클릭했을 때, **When** 인증 상태가 확인되면, **Then** 로그인 필요 안내 또는 로그인 페이지 이동이 제공된다.

---

### User Story 4 - 빈 결과와 반응형 사용성 (Priority: P2)

> 구직자는 조건에 맞는 공고가 없어도 현재 상태를 이해하고 다시 탐색할 수 있으며, 모바일에서도 동일한 기능을 사용할 수 있다.

**Acceptance Scenarios**:
1. **Given** 검색과 필터 조합에 일치하는 공고가 없을 때, **When** 목록 영역이 렌더링되면, **Then** 빈 상태 메시지와 조건 완화 안내가 표시된다.
2. **Given** 모바일 화면에서 사용자가 목록 페이지에 진입했을 때, **When** 공고 카드와 필터 패널을 확인하면, **Then** 텍스트가 겹치지 않고 한 열 레이아웃으로 사용할 수 있다.
3. **Given** 사용자가 페이지 하단으로 스크롤했을 때, **When** 상단 이동 버튼을 클릭하면, **Then** 페이지 최상단으로 부드럽게 이동한다.

## Edge Cases

- 검색어가 공백만 포함된 경우 전체 검색으로 처리할 것인가?
- 기간 필터에서 기준 날짜는 서버 현재일과 데이터의 최신 게시일 중 무엇을 사용할 것인가?
- 경력 조건이 `경력무관`, `신입`, `3~20년`처럼 범위가 다른 경우 교집합 판단을 어떻게 적용할 것인가?
- API 응답은 성공했지만 `items`가 빈 배열인 경우와 API 오류를 어떻게 구분해서 표시할 것인가?
- 스크랩 토글 요청이 실패했을 때 낙관적으로 바뀐 버튼 상태를 어떻게 되돌릴 것인가?
- 원본 공고 URL이 없는 공고를 상세로 열었을 때 원본 이동 버튼은 어떤 fallback을 사용할 것인가?

## Requirements

### Functional Requirements

- **FR-001**: 사용자는 채용 공고 목록 페이지에서 공개 채용 공고 카드를 확인할 수 있어야 한다.
- **FR-002**: 공고 카드는 회사명, 공고 제목, 직무/경력/채용 유형/지역 요약, 태그, 마감 정보, 출처, 조회수, 추천 여부, 스크랩 상태를 표시해야 한다.
- **FR-003**: 사용자는 회사명, 공고명, 직무, 태그, 기술 스택, 출처를 대상으로 키워드 검색을 수행할 수 있어야 한다.
- **FR-004**: 사용자는 직무, 경력, 채용 유형, 지역, 기업 규모 필터를 조합해서 목록을 좁힐 수 있어야 한다.
- **FR-005**: 사용자는 `오늘`, `7일`, `30일`, `기간 전체` 기준으로 게시 기간을 필터링할 수 있어야 한다.
- **FR-006**: 사용자는 `추천순`, `최신순`, `조회순` 중 하나로 목록을 정렬할 수 있어야 한다.
- **FR-007**: 활성화된 필터는 칩으로 표시되고, 사용자는 칩을 클릭해 개별 필터를 초기화할 수 있어야 한다.
- **FR-008**: 사용자는 공고 카드 또는 키보드 Enter/Space 입력으로 상세 드로어를 열 수 있어야 한다.
- **FR-009**: 사용자는 목록 카드에서 공고를 스크랩하거나 스크랩 해제할 수 있어야 한다.
- **FR-010**: 조건에 맞는 공고가 없을 때 빈 결과 메시지를 표시해야 한다.
- **FR-011**: 목록 페이지는 데스크톱, 태블릿, 모바일 화면에서 텍스트 겹침 없이 사용할 수 있어야 한다.
- **FR-012**: 검색, 필터, 정렬 변경 후 목록 개수와 화면에 보이는 카드 수는 일치해야 한다.
- **FR-013**: API 연동 시 목록 조회와 스크랩 요청은 `frontend/src/user/api` 하위 API 모듈을 통해 호출해야 하며 페이지 컴포넌트에서 직접 HTTP 호출을 작성하지 않아야 한다.
- **FR-014**: API 응답은 `ApiResponse<T>`의 `success`, `statusCode`, `message`, `data` 필드 기준으로 처리해야 한다.

### Key Entities

- **JobNoticeSummary**: id, company, title, jobType, experience, employmentType, location, companySize, salary, deadline, postedAt, tags, source, recommended, recommendScore, views, bookmarked, originalUrl, stacks
- **JobNoticeFilter**: keyword, jobType, experience, employmentType, location, companySize, period, sort
- **JobNoticeListStats**: totalOpenCount, todayNewCount, todayNewDelta, todayNewRate
- **BookmarkState**: jobNoticeId, bookmarked

## Success Criteria

- **SC-001**: 사용자는 페이지 진입 후 2초 이내에 첫 목록 또는 로딩 상태를 확인할 수 있다.
- **SC-002**: 검색어와 필터를 조합했을 때 화면의 공고 수와 결과 헤더의 공고 수가 항상 일치한다.
- **SC-003**: 공고 카드의 북마크 버튼 클릭은 카드 상세 열기 이벤트를 발생시키지 않는다.
- **SC-004**: 키보드만으로 검색 입력, 필터 선택, 정렬 선택, 공고 상세 열기, 스크랩 토글을 수행할 수 있다.
- **SC-005**: 모바일 360px 너비에서도 주요 텍스트와 버튼이 서로 겹치지 않는다.

## Assumptions

- v1 구현은 현재 `JobNoticeListPage.tsx`의 mock 데이터 구조를 기준으로 시작하되, API 연동 시 `api-spec.json`의 응답 구조로 전환한다.
- API 연동 전환 시 서버 상태는 TanStack Query 사용을 우선 검토하고, 검색/필터/정렬처럼 화면에 종속된 입력 상태는 React local state로 관리한다.
- 상세 페이지는 별도 라우트가 아니라 목록 페이지 안에서 `JobNoticeDetail` 드로어로 열린다.
- 목록 조회는 비로그인 사용자에게도 허용하지만, 스크랩 저장과 해제는 로그인 사용자만 가능하다.
- 추천순은 서버 연동 전에는 `recommendScore` 내림차순으로 처리한다.
- 최신순은 서버 연동 전에는 `postedAt` 또는 공고 ID 기준 내림차순으로 처리한다.
