# Checklist: 채용 공고 리스트

> tasks.md가 "무엇을 만들지"라면, 이 파일은 "제대로 만들어졌는지" 검증한다. 구현 완료 후 PR 전 본인이 직접 체크한다.

## Phase 1 - 계약 및 타입

- [ ] `api-spec.json`의 목록 응답 필드와 `JobNoticeTypes.ts`의 화면 필드가 매핑되어 있다.
- [ ] API 응답 계약이 `ApiResponse<T>`의 `success`, `statusCode`, `message`, `data` 필드를 기준으로 작성되어 있다.
- [ ] 목록과 상세가 같은 공고 ID와 스크랩 상태를 공유한다.
- [ ] 필터와 정렬 값은 화면 라벨, API 쿼리 값, 내부 상수 사이의 변환 규칙이 명확하다.
- [ ] `전체` 필터는 API 요청에서 생략하거나 서버가 전체 조건으로 해석한다.

## Phase 2 - 목록 조회

- [ ] 최초 진입 시 로딩 또는 목록이 사용자에게 명확히 표시된다.
- [ ] 목록/상세/스크랩 API 호출은 `frontend/src/user/api` 하위 모듈을 통해 수행된다.
- [ ] 페이지 또는 컴포넌트에서 `axios`나 `fetch`를 직접 호출하지 않는다.
- [ ] API 연동 시 서버 상태는 TanStack Query 사용을 우선 검토했거나, 사용하지 않은 이유가 남아 있다.
- [ ] API 성공, 빈 결과, 오류 상태가 서로 다른 UI 메시지를 가진다.
- [ ] 결과 헤더의 공고 수와 실제 카드 개수가 일치한다.
- [ ] 통계 카드의 전체 공고 수와 오늘 신규 공고 수가 API 또는 mock 응답 기준으로 표시된다.

## Phase 3 - 탐색 조건

- [ ] 검색 버튼 클릭과 Enter 입력이 같은 결과를 만든다.
- [ ] 인기 검색어 클릭 시 검색 input 값과 목록 결과가 함께 갱신된다.
- [ ] 직무, 경력, 채용 유형, 지역, 기업 규모 필터가 조합 조건으로 동작한다.
- [ ] 기간 필터 `오늘`, `7일`, `30일`, `기간 전체`이 게시일 기준으로 동작한다.
- [ ] `추천순`, `최신순`, `조회순` 정렬이 선택값에 맞게 목록 순서를 바꾼다.
- [ ] 활성 필터 칩 삭제 시 해당 조건만 초기화된다.

## Phase 4 - 카드 및 상세 액션

- [ ] 카드 클릭 시 선택한 공고의 상세 드로어가 열린다.
- [ ] 카드에서 Enter/Space 키를 눌러도 상세 드로어가 열린다.
- [ ] 북마크 버튼 클릭은 상세 드로어를 열지 않는다.
- [ ] 스크랩 성공 후 목록 카드와 상세 드로어의 북마크 아이콘 상태가 동일하다.
- [ ] 스크랩 실패 시 이전 상태로 되돌아가며 실패 안내가 제공된다.
- [ ] 상세 드로어 닫기 버튼, 오버레이 클릭, Escape 키 닫기가 모두 동작한다.

## Phase 5 - UI, 접근성, 검증

- [ ] 360px 모바일 화면에서 검색, 필터, 카드, 정렬, 버튼 텍스트가 겹치지 않는다.
- [ ] 720px 이하에서 공고 카드는 한 열로 표시된다.
- [ ] 1080px 이하에서 필터 패널은 본문 위 또는 한 열 흐름으로 자연스럽게 배치된다.
- [ ] 모든 아이콘 버튼은 `aria-label` 또는 보이는 텍스트로 목적을 알 수 있다.
- [ ] 키보드만으로 검색, 필터, 정렬, 카드 진입, 스크랩 토글, 상세 닫기를 수행할 수 있다.
- [ ] 상단 이동 버튼은 페이지 최상단으로 부드럽게 이동한다.
- [ ] 프로젝트 표준 빌드 또는 타입 검증 명령이 통과한다.
