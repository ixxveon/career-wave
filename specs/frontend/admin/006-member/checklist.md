# Checklist: 회원관리 (Member Management)

> `spec.md`가 "무엇을 만들지"라면, 이 파일은 "제대로 만들어졌는지" 검증한다.
> UI 구현 완료 기준 체크 + API 연동 전 재검증 항목을 포함한다.

---

## Phase 1 — ERD 정합성

- [ ] `Member` 인터페이스의 `status` 값이 ERD 기준 `ACTIVE / SUSPENDED / BANNED`로 정렬되어 있다.
- [ ] `SuspendDuration` 값이 ERD 기준 `THREE_DAYS / SEVEN_DAYS / THIRTY_DAYS / PERMANENT`로 정렬되어 있다.
- [ ] `role` 필드가 `ROLE_USER / ROLE_COMPANY`만 사용한다 (`ROLE_MENTOR` 등 ERD 미존재 값 없음).
- [ ] `Member` 인터페이스에 ERD에 없는 컬럼(`position`, `permission_level` 등)이 포함되지 않는다.
- [ ] `CompanyMember`의 사업자등록번호 필드명이 ERD 컬럼명(`certificate_number`)과 매핑 주석으로 명확하다.
- [ ] `suspend_histories`의 `sanction_type` 값이 `WARNING / SUSPEND / BLACKLIST`로 정렬되어 있다.
- [ ] 기업 회원 `hrStatus`의 현재 한국어 값과 API 연동 후 ERD enum(`PENDING / ACTIVE / REMOVED`) 매핑이 `spec.md`에 문서화되어 있다.

---

## Phase 2 — 개인 회원 탭

- [ ] KPI 카드 4종(전체 회원 수·오늘 신규·프리미엄 구독·정지 회원 수)이 표시된다.
- [ ] 이름·이메일·로그인 ID 키워드 검색이 실시간으로 작동한다.
- [ ] 목록 테이블에 번호·이름·이메일·로그인ID·회원유형·플랜·가입일·상태·관리 컬럼이 표시된다.
- [ ] 회원유형 배지가 `ROLE_USER → 개인`, `ROLE_COMPANY → 기업`으로 표시된다.
- [ ] 플랜 배지가 `FREE / PREMIUM` 구분 스타일로 표시된다.
- [ ] 상태 배지가 `ACTIVE → 정상 / SUSPENDED → 정지 / BANNED → 영구정지`로 표시된다.
- [ ] 체크박스 전체 선택/해제가 작동한다.
- [ ] 테이블이 가로 스크롤(`tableScroll`)로 처리되어 화면이 좁아도 깨지지 않는다.

---

## Phase 3 — 개인 회원 상세 모달

- [ ] 상세 모달에 권한·구독 플랜·가입일·최근 접속·현재 상태·신고 횟수·경고 횟수가 표시된다.
- [ ] 경고 횟수 dot indicator가 `warningCount` 값에 맞게 채워진 점/빈 점으로 표시된다.
- [ ] 경고 횟수 dot 개수가 `WARN_THRESHOLD(3)` 기준으로 렌더링된다.
- [ ] `warningCount >= WARN_THRESHOLD`일 때 "활동정지 권고" 배지가 표시된다.
- [ ] `warningCount === WARN_THRESHOLD - 1`일 때 "1회 추가 시 활동정지 권고" 안내가 표시된다.
- [ ] 모달 외부 클릭 또는 닫기 버튼으로 모달이 닫힌다.
- [ ] 상세 모달에서 "활동 정지" 버튼 클릭 시 활동 정지 모달로 전환된다.

---

## Phase 4 — 활동 정지 모달

- [ ] 정지 기간 버튼 4종(3일/7일/30일/영구)이 표시된다.
- [ ] 기간 선택 시 선택된 버튼이 강조(진한 배경) 표시된다.
- [ ] `PERMANENT` 선택 시 정지 처리 확정 버튼이 붉은 계열 색상으로 변경된다.
- [ ] 기본 선택 기간이 `SEVEN_DAYS(7일)`로 설정된다.
- [ ] 정지 사유 텍스트에어리어에 입력이 가능하다.
- [ ] 처리 관리자 ID 영역이 표시된다.
- [ ] 정지 처리 후 모달이 닫힌다.

---

## Phase 5 — 기업 회원 탭

- [ ] 승인 대기 건이 있을 때 탭 뱃지가 표시된다.
- [ ] KPI 카드 4종(전체·승인 대기·승인 완료·반려)의 숫자가 실제 데이터와 일치한다.
- [ ] 담당자명·기업명·이메일 키워드 검색이 실시간으로 작동한다.
- [ ] 승인 상태 필터(전체/승인 대기/승인 완료/반려)가 작동한다.
- [ ] 목록 테이블에 회원ID·HR 담당자명·이메일·기업명·사업자등록번호·재직증명서·가입일·승인 상태·관리 컬럼이 표시된다.
- [ ] 재직증명서 파일명이 18자 초과 시 말줄임(`…`)으로 표시된다.
- [ ] 승인 대기 건에만 "승인" 버튼이 목록에 표시된다.

---

## Phase 6 — 기업 회원 상세 및 승인·반려 모달

- [ ] 상세 모달에 HR 담당자명·이메일·기업명·사업자등록번호·가입일·승인 상태·재직증명서가 표시된다.
- [ ] 반려 사유가 있는 경우 별도 영역에 표시된다.
- [ ] 승인 대기 상태에서만 "승인 처리" / "반려 처리" 버튼이 표시된다.
- [ ] 승인 확인 모달에 담당자명·기업명·즉시 이용 가능 안내가 표시된다.
- [ ] 반려 처리 모달에 사유 입력창과 "신청자에게 안내됩니다" 안내가 표시된다.
- [ ] 승인 확정 버튼이 강조색(진한 파란 계열)으로 표시된다.
- [ ] 반려 처리 버튼이 위험색(붉은 계열)으로 표시된다.

---

## Phase 7 — API 연동 전 체크

- [ ] `Member` 인터페이스 필드가 백엔드 응답 DTO와 매핑 가능한 구조인지 확인한다.
- [ ] 개인 회원 목록 API 쿼리 파라미터(`role`, `status`, `plan`, `keyword`, `page`, `size`)와 현재 필터 상태가 연결 가능한 구조인지 확인한다.
- [ ] 기업 회원 `hrStatus` 한국어 값을 ERD enum(`PENDING/ACTIVE/REMOVED`)으로 전환하는 변환 함수가 필요한지 확인한다.
- [ ] 정지 처리 API 요청 바디(`sanction_type`, `duration`, `reason`)가 현재 모달 상태와 매핑 가능한지 확인한다.
- [ ] 기업 회원 반려 사유 미입력 시 빈 값 전송 방지 유효성 검증이 추가되어 있는지 확인한다.
- [ ] 처리 관리자 ID가 세션에서 가져오도록 전환되었는지 확인한다.
- [ ] API 성공·실패 응답이 `ApiResponse<T>` 형식(`success`, `statusCode`, `message`, `data`)을 기준으로 처리되는지 확인한다.
- [ ] 모든 HTTP 호출이 `frontend/src/admin/api` 또는 팀이 정한 API 계층을 통해 수행되는지 확인한다.
